## 전략 1: 구현
- 아무리 뒤의 작업이 완료되어도, 앞의 작업이 배포되어야 하므로, 큐로 생각할 수 있음
- 실제로 개발 과정을 시뮬레이션
    - 바깥 while문 루프 한 번은 하루와 같음: 각 작업 진도율에 각 속도를 더함
    - 그 날의 배포 시점에 맨 앞의 작업을 지금 배포할 수 없는 작업이 나올 때까지 맨 앞을 pop함
        - ⚠️ **작업을 pop할 때, 진도뿐만 아니라 속도 리스트에서도 pop해야 함!**
        - 개선점: `pop(0)`의 시간 복잡도가 `O(N)`이므로,
            - `deque`를 써서 `popleft()` 하거나
            - 리스트를 `reverse` 해서 `pop()` 하는 방식으로 개선
    - 그 날에 배포한 작업이 있으면, 정답 리스트에 그 작업 개수를 append
- 바깥 while문이 끝나면, 정답 리스트를 반환

## 전략 2 (🗒️)
- 각 기능이 배포되기까지 필요한 일수를 계산
    - 잡기술: `ceil` 없이 `음수 // 양수` 후 부호 변환하여 올림 효과를 구현할 수 있음
        - `음수 // 양수`: `음수 / 양수`에서 절댓값을 올림한 것과 같음
- 배포 그룹을 만듦
    - 큐가 비어있거나, 배포 큐의 맨 뒤 작업의 필요한 일수보다 현재 작업의 배포에 필요한 일수가 더 많으면, 새로운 그룹을 만듦
    - 그렇지 않으면, 배포 큐 맨 뒤의 작업 그룹에 포함
